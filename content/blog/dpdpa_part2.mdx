## Part 2: What Counts as Personal Data

DPDPA applies only to digital personal data.

This looks narrow. It is not.

Any data that can identify an individual—directly or indirectly—is in scope once it is digitized.

This includes obvious identifiers like names and phone numbers. It also includes combinations of fields that can be linked back to a person.

The boundary is not the field. It is identifiability.

### Where Companies Misread Scope

Most teams think in terms of structured databases.

That is incomplete.

Personal data exists across:

* Application databases  
* Logs and analytics tools  
* Spreadsheets and exports  
* Email attachments  
* CRM systems  
* Third-party SaaS tools  

If the data is digital and tied to a person, it is covered.

### The Hidden Surface Area

The highest risk does not sit in primary systems.

It sits in:

* Data exports shared over email  
* Temporary CSVs used for analysis  
* Debug logs capturing user information  
* Backups retained indefinitely  
* Vendor systems with partial datasets  

These are rarely mapped. Almost never controlled.

### Direct vs Indirect Identification

Direct identifiers:
* Name  
* Email  
* Phone number  
* Government IDs  

Indirect identifiers:
* IP address  
* Device identifiers  
* Location data  
* Behavioral patterns  

Individually, indirect identifiers may seem harmless. Combined, they become personal data.

### The Real Constraint

DPDPA does not regulate categories. It regulates usage.

If a dataset can identify a person, then:

* It must have a defined purpose  
* That purpose must be enforced  
* It must be deleted when no longer needed  

Classification without enforcement is irrelevant.

### What This Means Operationally

You cannot protect what you cannot see.

A compliant system requires:

* A complete inventory of where personal data exists  
* Mapping of how it flows between systems  
* Visibility into which vendors receive it  
* Control over how long it is retained  

Without this, scope is undefined.

### Where Most Teams Fail

* Treating only primary databases as in scope  
* Ignoring logs, backups, and exports  
* Assuming anonymization without verification  
* Not tracking data movement across vendors  

This leads to false confidence and real exposure.

### The Only Useful Model

Stop thinking in terms of systems.

Start thinking in terms of data units:

Each unit of personal data must have:

* Origin (where it came from)  
* Purpose (why it exists)  
* Flow (where it goes)  
* Expiry (when it is deleted)  

If any of these are unknown, the data is uncontrolled.

DPDPA expands scope by removing ambiguity.

If data can identify a person, it is your responsibility.
