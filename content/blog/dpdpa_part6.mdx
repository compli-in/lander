## Part 6: Obligations That Actually Break Teams

DPDPA defines a clear set of obligations for Data Fiduciaries.

Most teams understand them conceptually.

They fail in execution.

### The Core Obligations

At a minimum, every Data Fiduciary must ensure:

* Purpose limitation  
* Data minimization  
* Data accuracy  
* Reasonable security safeguards  
* Time-bound data erasure  
* Breach notification  

These are not independent requirements.

They are interdependent system constraints.

### Purpose Limitation

Data must only be used for the purpose it was collected for.

Failure pattern:

* Purpose defined broadly or vaguely  
* Data reused across functions without restriction  
* No enforcement at the system level  

If purpose is not enforced, it does not exist.

### Data Minimization

Only the data required for a specific purpose should be collected.

Failure pattern:

* Collecting excess fields “just in case”  
* No review of necessity over time  
* Data accumulation without deletion  

Excess data increases risk without adding value.

### Data Accuracy

Data must be accurate and kept up to date.

Failure pattern:

* No mechanisms for correction  
* Duplicate or inconsistent records  
* Downstream systems not updated  

Inaccurate data creates both compliance and operational risk.

### Security Safeguards

Organizations must implement reasonable security measures.

Failure pattern:

* Security treated as perimeter defense only  
* No control over internal access  
* Lack of monitoring and audit trails  

Security must extend to how data is accessed and used internally.

### Time-Bound Erasure

Data must be deleted once the purpose is fulfilled.

Failure pattern:

* No defined retention periods  
* Backups and logs retained indefinitely  
* Deletion not enforced across systems  

If data is not deleted, compliance is already broken.

### Breach Notification

Data breaches must be reported to the regulator and affected individuals.

Failure pattern:

* No centralized incident detection  
* Delayed identification of breaches  
* No defined notification workflow  

Breach response must be immediate and structured.

### Why Teams Break Here

These obligations require coordination across:

* Engineering  
* Legal  
* Security  
* Operations  

Most companies treat them in isolation.

That guarantees failure.

### The Only Useful Model

Translate obligations into system behavior:

* Purpose → access control and processing rules  
* Minimization → schema and collection limits  
* Accuracy → update and sync mechanisms  
* Security → access logs and monitoring  
* Retention → automated deletion  
* Breach → detection and escalation pipelines  

If obligations are not encoded into systems, they are not real.

DPDPA does not test awareness.

It tests whether systems behave correctly under pressure.
